Metadata Objects


Overview of Objects

The core of PODD is the capability to manage metadata and data at the object level. Objects can be considered as a general "thing" for which data or metadata can be captured. e.g.,

  • a Project object;
  • an Investigation object;
  • a Material object;
  • a Measurement object;
  • a Data object.

Concept Objects and Concrete Objects are defined in PODD. A Concept Object is an OWL (Web Ontology Language) class from which concrete objects are instantiated. It is written in OWL/RDF (Resource Description Framework). Concept objects are maintained as an OWL file.

Concrete objects are instances of a concept object. A Project is a concept object, objects of the project are concrete objects, such as Investigation Objects, Material Objects etc.


The Parent-Child Relationship

PODD object relationships are governed by Subject-Predicate-Object rules defined in OWL/RDF. The predicate determines the nature of the relationship between objects.

The Parent-Child relationship is where a new object is added to PODD as a child of an existing object. The relationship is identified by the predicate selected, e.g.,

  • Project A (subject) hasProjectPlan (predicate) Project Plan B (object).
  • Genotype X (subject) hasGene (predicate) Gene Y (object).

All objects with the exception of Project objects cannot exist in isolation. When instantiating concrete objects the appropriate predicate must also be selected. Consequently objects will be created in relation to their parent object.

The Object Hierarchy

The PODD data model allows for data and metadata to be grouped into a DAG (directed acyclic graph) like structure, with a project object as the highest possible node.

Firgure 6.2 – Screen shot of an object hierarchy in a View Project Object page

Figure 6.2 shows a screen shot of the object hierarchy of PODD project (Gav’s Project).  The Hierarchy section is displayed at the end of each View Object page.

The hierarchy section displays the parent and direct children of the current object. It means only the existing children of the projects are displayed. All existing children would be listed. The next level children are not listed, only the number of objects is displayed at the end. It does not display any nodes above the current parent either.

The Hierarchy display is semi-interactive. The user may navigate to an object's view by clicking on the object title in the hierarchy display (The titles of the child objects are hyperlinked.).

The Hierarchy will display the current parent object, its Title, Type and Number of Children. It will not display any nodes above the current parent.
The Hierarchy will display entries for:

  • The current object's Parent Object;
  • The current object;
  • The current object's children;
  • Action Buttons

Based to the user’s repository and project specific role, different action buttons will be displayed at the bottom of the page: Edit Object, Paste, Add Child Object, Delete and Publish Project (see the table below).

  • An Edit Object button takes the user to Edit Project Object page. The user can edit the project details, update the project participants, add or delete data items. The same action would be effected by clicking on the edit tab.
  • An Add Child button would take the user to an Add Child Object page, with the project as the parent.
  • A Paste button takes the user to the clipboard screen. The user can select objects/predicates to paste to the Project if there are any previously copied objects/predicates available on the screen.
  • A Delete button allows the authorised user to delete the project if there is no child object or all child projects have been already deleted.
  • A Publish Project button is displayed on the View Project Object page only when the Project status has been set to Complete.

The Object Hierarchy can also be viewed in the Browser Pane, the whole object hierarchy can be listed when displayed in an expanding state (details will be described later).


Object References

Besides the Parent-Child relationship, objects can have another form of relationship to each other—Object references.

Object references describe the on-screen functionality for selecting objects to be referenced by the current object. e.g.

  • an Analysis Object may refer to a Platform object
  • a Data object may refer to a Material object
  • an Event object may refer to a Protocol object.

Object references are displayed as Refers To tables on the View Project Object page. Where the subject is the object being referenced from, a "Refers To" relationship will look up to the project level and then down to every level except the lowest level Investigation branch that the subject is on. A "Refers To" relationship will not look at any branches below the branch the subject is on.
