PODD User's Roles


Repository User Roles

A user will have a role that will assist in determining what functions they can perform and what access they will have to existing projects.

There are four distinct categories of users for the PODD website. Each role builds on the privileges of the previous one. They are

  • Public users;
  • Registered Users;
  • Project Creator;
  • Repository Administrator.

Public users

Public user is also known as anonymous user. They are not registered or logged in. This role has the least privilege. Privileges include:

  • Can view public acess projects;
  • Can search;
  • Can register to the system.

Registered Users

Registered user is also known as repository user. This is a standard user who is registered to the system. Privileges include:

  • Can view public access projects;
  • Can maintain their own user profiles;
  • Can search and filter projects;
  • Can access projects based on project role.

Project Creator

Project creators are registered users with additional ability of being able to create projects.

  • Can view public access projects;
  • Can maintain their own user profiles;
  • Can search and filter projects;
  • Can access projects based on project role
  • Can create projects (Note that they will not have any further access over the project unless they define themselves with a project role during project creation. The default behaviour of the system is to define the Project Creator with a Project Administration role).

Repository Administrator

Repository Administrators are facility staff members who administer PODD clients and PODD contents.

  • Can Create, Read, Update and Delete repository wide objects;
  • Can Create, Update and Delete users;
  • Can administer project roles;
  • Can search and filter projects;
  • Can copy and paste objects between projects;
  • Can restore deleted objects;
  • Can publish/unpublish projects.


Project Specific Roles

Every project must have one or more project participants. These roles define access rights over the contents of un-published projects. They are defined as four different project specific roles:

  • Project Administrator;
  • Principal Investigator;
  • Project Member;
  • Project Observer.

Project Administrator

An administrator of a particular project (all access rights), including

  • Can Create, Read, Update and Delete project objects (CRUD);
  • Can copy objects between projects;
  • Can publish projects;
  • Can administer project roles.

Principal Investigator

Is identified as the PI and thus responsible for the project contents:

  • Can perform all project administrator functions (CRUD);

Project Member

Member of a particular project:

  • Can Create, Read and Update project objects (CRU).

Project Observer

An observer of the project that only has READ access:

  • Can read project objects (READ).
