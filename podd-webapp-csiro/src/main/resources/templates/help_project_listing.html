Listing Projects


Overview

A Project Object is considered the foundation object for any projects in PODD. That is, objects will exist in a hierarchy of relationships with a Project object as the top node. For this reason the first view that a user will see is the Projects tab with a list of projects that they may access.


Actors

All users, whether registered or public access, will be able to list projects. Three types project are defined in the system.  They are:

  • My Projects: Projects the user is associated with (see the definition at the end of this section).
  • Public Projects: Public access projects (published).
  • All Projects: all projects in the system, including deleted projects. All projects are only available to the Repository Administrator.

Different types of projects will be listed based on the user's repository roles:

  • Public users can only list Public Projects.
  • Registered users/Project Creators can list My Projects and Public Projects.
  • Administrators can list All Projects in the PODD instance.

* A user is associated with a project if, they are defined as either an Project Administrator/Principal Investigator (CRUD access), Project Member (CRU) or Project Observer (R) of the project.


The Project Pane

When a user navigates to the PODD website without loging in, the default view is the Projects Pane. All Public access projects are listed in the Project Pane.

Once the user logs in, the Project Pane will list all projects the user is associated with (My Projects) and all public access projects (Public Projects).

 

Figure 8.1 – Default Listing Projects View in Projects Tab for a regestered user

As shown in Figure 8.1, for each Project object listed in the Project Pane the following information is available:

  • The Project Title (Links to Browser View of the project);
  • The user's project specific role (in brackets), will be either:
    • Project Administrator
    • Principal Investigator
    • Project Member
    • Project Observer
    • Public access
  • The Principal Investigator, prefaced by the term Principal Investigator (it won't be repeated if the user is the Principal Investigator);
  • The Lead Institution, prefaced by the term Lead Institution;
  • Project description. First 200 charactors, whole words (no breaks in words) or to first line return, which ever comes first;
  • A Browse Project Link, which takes the user to the Browser View of the project (same as the link behind the project title).

My Projects and Public Projects are separated in the view. Where a user is associated with a public access project, this project would be displayed under the heading My Projects and not Public Projects.

A Repository Administrator will, by default, have a listing of All Projects. If a Repository Administrator wants to separate My Projects, Public Proects or any other Projects, he can filter the project by clicking the Filter Projects button on the Project Pane.

Both My Projects and Public Projects can be hidden by clicking the collapse icon next to the headings (see Figure 8.1). All Projects can be hidden to the administrators as well.


Action Buttons

There are also additional buttons displayed for different user roles:

  • Filter Projects button is visible to all registered users. It will take user to the filter function (see Filtering Projects for details).
  • Create Projects button is only visible to Project Creator and Repository Administrator. It will take user to the Create Project function (see Creating a Project).  

Pagination

Where a list of projects exceeds the on screen maximum (20) a list of page numbers will be displayed at the bottom for the user to page through the project list. This pagination function will exist for both My Projects and Public Projects. So if there are more than 20 My Projects and the user selects the next page then the My Projects list would be updated and not the Public Projects list, and vice versa for Public Projects.
